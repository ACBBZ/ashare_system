# 主板短线总控工作流

这是一个面向**主板短线交易体系**的总控 skill。

它不重复造轮子，而是把现有 skills 按交易流程编排成一条完整链路：

1. `ashare-monitor-cron` —— 盘中静默采集、收盘摘要、候选池初筛
2. `stock-monitor-anomaly` —— 异动扫描、候选池分层
3. `stock-analysis-cn` —— 单票技术面 / 支撑压力 / 执行建议
4. `stock-deep-analysis-cn` —— 单票完整交易型深度分析
5. `finance-news-cn` —— 新闻与催化核验
6. `ashare-position-watch-obsidian` —— 持仓股 / 候选股盘后执行分析与 Obsidian 落地
7. `akshare-open` —— 板块、资金、事件、公告、估值、宏观等综合投研底座
8. `akshare-stock-data` —— 原始数据与接口兜底

核心目标不是“泛研究”，而是：
- 判断当前市场是否适合短线出手
- 找出强板块
- 从强板块中筛出低位趋势成形、且有催化和承接的主板股
- 形成可执行的次日计划
- 对持仓股和候选股做盘后执行层分析

## 默认规则（除非用户明确覆盖）

### 市场范围
- 个股候选与次日计划**只保留主板股票**：`600/601/603/605/000/001/002/003`
- 默认排除：`300/301`（创业板）、`688/689`（科创板）、`8/4` 开头（北交所等）
- ETF/LOF 等场内基金产品允许纳入观察，但必须**单独分组输出**，不要与主板个股混排

### 用户风格
- 默认按**小资金短线**视角输出
- 优先聚焦 1~2 个最强核心方向
- 强调：
  - 盈亏比
  - 计划买点
  - 确认条件
  - 放弃条件
  - 失效条件
- 不输出大资金分散配置式建议
- **新增执行约束**：先做硬过滤，再决定是否输出可下单模板；不要把观察票包装成可执行票

### 候选分层（新增强制规则）
所有候选默认分三层：
- **A层：可执行池**
  - RR >= 1.5
  - 非退潮板块，优先 主升 / 修复
  - 流动性达标（至少避免明显冷门、低成交额）
  - 适合账户体量（至少一手成本不能明显超出账户可承受范围）
- **B层：观察池**
  - 逻辑还行，但 RR / 流动性 / 板块阶段 / 价格结构有一项不够硬
  - 只保留观察结论，不输出完整可下单模板
- **C层：仅记录**
  - RR < 1，或板块退潮，或流动性明显不足，或账户体量明显不匹配
  - 只记录，不进入次日执行计划

### 假精细控制（新增）
- 只有 **A层** 才输出完整的开盘前 / 09:26 可执行模板
- **B层** 只输出：
  - 为什么观察
  - 需要补什么确认
  - 何时升级到 A 层
- **C层** 只输出剔除原因
- 如果当天 **A层为空**，应明确写：
  - 今天默认只观察
  - 不为交易而交易
  - 不生成“看起来很细但其实不能下单”的模板

### 候选股标准
优先选择符合以下条件的股票：
1. 属于强板块 / 主线板块
2. 板块阶段优先：`主升 > 修复 > 轮动 > 分歧 > 退潮`
3. 个股角色清晰：`龙头 / 中军 / 补涨 / 跟风`
4. 处于**低位趋势成形**，而不是纯高位情绪冲顶
5. 存在可核验催化：公告、业绩、订单、政策、机构调研、回购、股东变化等
6. 有承接，不是单纯脉冲拉升

## 何时使用

当用户提出以下任一类任务时优先使用本 skill：
- “按我的体系做一份收盘复盘和次日计划”
- “帮我按主板短线框架筛候选股”
- “今天市场环境适不适合做短线？”
- “找出强板块里低位趋势成形、有催化和承接的票”
- “把收盘摘要、候选池、持仓分析串成完整流程”
- “按我当前主板短线规则做一遍完整工作流”

## 如何使用

### 用法 1：完整收盘复盘
当用户要“今天完整复盘 + 明日计划”时：
1. 先按 `ashare-monitor-cron` 读取本地盘中数据与收盘口径
2. 再按本 skill 的流程判断：市场环境 → 强板块 → 候选池 → 次日计划
3. 输出时优先参考：
   - `templates/close-summary-template.md`
   - `references/board-stage-guide.md`
   - `references/risk-and-failure-signals.md`

### 用法 2：只做候选池筛选
当用户要“帮我筛明天候选股”时：
1. 先用 `stock-monitor-anomaly` 找主板异动股 / 强板块成分股
2. 再用 `finance-news-cn` 补催化
3. 再用 `stock-analysis-cn` 补趋势与承接
4. 输出时优先参考：
   - `templates/candidate-pool-template.md`
   - `references/candidate-scoring.md`
   - `references/risk-and-failure-signals.md`

### 用法 3：只分析一只票
当用户要“看一只票值不值得做”时：
- 轻量分析走 `stock-analysis-cn`
- 深度分析走 `stock-deep-analysis-cn`
- 但结论仍应回到本 skill 的统一口径：
  - 板块
  - 板块阶段
  - 个股角色
  - 趋势结构
  - 催化
  - 承接
  - 支撑压力
  - 确认 / 放弃 / 失效条件

### 用法 4：盘后持仓 / 候选执行分析
当用户要“持仓股和候选股明天怎么处理”时：
1. 先读最近的收盘摘要和候选池结果
2. 再调用 `ashare-position-watch-obsidian`
3. 输出趋势、通道、支撑压力、盈亏比和操作策略

> 这个总控 skill 不负责替代所有底层 skill，而是负责：在正确的时机调用正确的 skill，并把最终结论统一到同一套短线主板框架里。

## 总控流程（强制顺序）

### 0. 先识别任务类型
先判断当前任务属于哪一类：

#### A. 收盘复盘 / 次日计划
输出重点：
- 市场总览
- 强板块
- 候选池
- 次日计划
- 自我复盘留空

#### B. 候选池筛选
输出重点：
- 异动股 / 强板块成分股
- 候选分层
- 是否值得继续跟踪

#### C. 单票分析
输出重点：
- 所属板块
- 板块阶段
- 技术结构
- 催化
- 承接
- 支撑压力
- 盈亏比
- 操作策略

#### D. 持仓 / 候选盘后执行分析
输出重点：
- 今日收盘分析
- 明日走势预判
- 通道
- 支撑 / 压力
- 盈亏比
- 操作策略
- Obsidian 落地

## 1. 市场环境判断
优先调用：`ashare-monitor-cron` + `akshare-open`

至少判断：
- 大盘涨跌
- 两市成交额
- 涨跌停数量
- 连板高度
- 市场风格：趋势 / 连板 / 轮动 / 退潮
- 当前短线环境：顺风 / 中性 / 逆风

要求：
- 先判断“能不能做”，再判断“做什么”
- 如果环境偏逆风，结论中必须显式降低激进度
- **新增**：尽量把“主升 / 修复 / 轮动 / 退潮”改成硬规则打分，而不是纯主观表述。最低建议规则：
  - 主升板块数量
  - 修复板块数量
  - 退潮板块数量
  - 涨停 / 跌停对比
  - 最高连板
  - 量能变化
- 输出里最好带：
  - 触发了哪些硬规则
  - 综合分
  - 当前最大总仓位建议

## 2. 强板块识别
优先调用：`akshare-open`

至少输出：
- 板块涨幅
- 板块资金流
- 板块龙头
- 板块内最高连板
- 板块阶段：主升 / 修复 / 分歧 / 退潮 / 轮动
- 板块逻辑催化
- 是否与大盘共振

要求：
- 板块最多展开 3 个，最多补充 2 个观察方向
- 不要做无边界遍历

## 3. 候选池筛选
优先调用：`stock-monitor-anomaly` + `finance-news-cn` + `stock-analysis-cn`

筛选流程：
1. 先做主板过滤
2. 再从强板块成分股和异动股中筛选
3. 对候选股补催化核验
4. 对入选股补趋势结构和承接分析

每只候选至少写：
- 名称 / 代码
- 所属板块
- 板块阶段
- 个股角色：龙头 / 中军 / 补涨 / 跟风
- 趋势结构
- 成交量 / 换手 / 承接
- 催化逻辑
- 是否符合“低位趋势成形 + 催化 + 承接”
- 明日关注点
- 失效条件

结果必须分层：
- **核心观察**
- **次级观察**
- **仅记录**

若当天需要跟踪 ETF/LOF：
- 单独开 `## 场内基金观察` 小节
- 不与主板候选股混排

## 4. 单票分析分流

### 轻量单票分析
优先调用：`stock-analysis-cn`

适用：
- “这只票值不值得关注？”
- “成本价在这里该怎么处理？”
- “支撑压力在哪？”

### 深度单票分析
优先调用：`stock-deep-analysis-cn`

适用：
- “全面分析这只票”
- “做一份完整交易型深度分析”
- “从板块、趋势、催化、资金、事件都看一下”

## 5. 新闻与催化核验
优先调用：`finance-news-cn`

规则：
- 优先核验硬催化：公告、业绩、订单、政策、机构调研、回购、股东变化
- 对同一事件尽量交叉核验 2 个以上来源
- 若未找到可信直接催化，必须明确写：`未找到可信直接催化`
- 不把传闻、论坛贴、二次转载当成等权依据

## 6. 收盘复盘与次日计划
优先调用：`ashare-monitor-cron`

固定结构建议：
1. 市场总览
2. 板块分析
3. 个股筛选（最重要）
4. 次日计划
5. 自我复盘（留空）

次日计划必须包含：
- 明天看好的 3 个板块
- 每个板块 1~3 个备选股
- 每只股的计划买点
- 每只股的确认条件
- 每只股的放弃条件
- 若适用：支撑位 / 压力位 / 盈亏比参考
- 风险提示

**新增执行要求：**
- A层候选才进入“次日执行模板”
- B层候选只写观察原因，不给细致下单计划
- C层候选只写剔除原因
- 若 A层为空，次日计划应明确写“默认只观察 / 不出手”

## 6.1 候选记分板（新增）
建议维护最近 **20~40 个交易日** 的候选跟踪统计，但实战上不要只做 1/3/5 日静态回看，应该升级成 **1~20 个交易日逐日跟踪**。

### 推荐记录口径（已验证更实用）
至少记录：
- trade_date
- code / name / sector / stage / role / tier
- entry_ref（**模拟买点**，不是简单用入选当日收盘价）
- entry_low / entry_high
- entry_trade_date（**从入选后的次一交易日开始**计入跟踪）
- days_tracked（当前已跟踪到第几日）
- current_ret
- best_ret_20 / worst_ret_20
- 逐日跟踪表：1~20 日每一日的 close/high/low 与 close_ret / best_ret / worst_ret

### 模拟买点的推荐生成方式
优先级应为：
1. 候选分析文件里的 **关键支撑位**
2. 候选分析文件里的 **明日预估波动区间**
3. MA5 / MA10 / MA20 形成的支撑带
4. 再退化到收盘价折价区

不要直接默认“入选当日收盘价 = 模拟买点”，因为这会明显扭曲短线策略验证。

### 统计输出建议
输出时至少汇总：
- A/B/C 层样本数
- A/B/C 层 **1 / 3 / 5 / 10 / 20 日**胜率
- A/B/C 层 **1 / 3 / 5 / 10 / 20 日**平均收益
- A/B/C 层 20 日最佳波动均值
- A/B/C 层 20 日最差波动均值

并在每日 `close-summary` 中额外显示：
- 每只重点跟踪股的模拟买点
- 入场日（次一交易日）
- 当前已跟踪天数
- 当前收益率
- 20 日内最佳 / 最差波动

### 工程实现经验（已验证有效）
- 推荐主表：`candidate_tracking`
- 推荐明细表：`candidate_tracking_daily`
- `candidate_tracking_daily` 用 `(trade_date, code, horizon_day)` 做主键，保存 1~20 每日轨迹
- 每日后台脚本只做两件事：
  1. 回填历史 `close-summary + 持仓股与候选股分析` 文件
  2. 更新已有候选到最新的 1~20 日轨迹
- 这类维护任务应改成 **后台本地 cron**，不要显式发消息给用户
- `close-summary` 生成时只读取 DB 汇总结果展示，不要临时重新做重型回测

### 历史回填经验
如果旧 `close-summary` 质量不足，只能提取到候选列表而拿不到推荐买点，应继续结合：
- 同日 `持仓股与候选股分析-*.md`
- 其中的关键支撑位 / 压力位 / 明日预判 / 盈亏比

这样才能把旧样本高质量补齐，让记分板真正可用，而不是只得到一堆“样本数有了、收益全缺失”的假统计。

用途：不是为了好看，而是为了验证：
- A层是否真的比 B层强
- 哪类板块 / 角色更适合用户的小资金短线
- 哪些过滤条件在拖后腿
- “推荐买点 + 次一交易日入场”的口径下，策略是否真的有正期望

## 7. 板块梯队复盘

新增必做环节，适用于**收盘复盘**与**持仓/候选执行分析**之间或并列。

优先数据源：`sector_constituent_snapshots`（来自 `ashare-background-monitor` 当天盘中采集）。

对每个强板块输出：
- **龙头**：板块综合最强领涨股（综合涨幅 + 成交额得分最高的成分股）
- **中军**：成交额大、板块代表性强的票
- **补涨**：跟随板块上涨但非龙头 / 中军的票

每只股票注明：
- 代码 / 名称
- 涨跌幅
- 成交额
- 所属板块阶段：主升 / 修复 / 轮动 / 退潮

并结合候选池状态，问：
- 龙头是否继续带队？
- 中军是否出现掉队迹象？
- 补涨股是否开始变弱、暗示板块可能轮动？

> 若当天盘中未采集到 `sector_constituent_snapshots`，则退化为从 `close-summary.md` 的板块分析文本中提取梯队信息，但仍须标注"数据来源：close-summary 文本（盘中成分缓存不可用）"。

## 8. 持仓股 / 候选股盘后执行分析
优先调用：`ashare-position-watch-obsidian`

适用：
- 收盘摘要之后
- 对持仓股和候选股做执行层分析

至少输出：
- 今日收盘分析
- 明日走势预判
- 上升 / 下降通道
- 支撑位 / 压力位
- RSI / MACD / ATR
- 盈亏比参考
- 操作策略

并写入 Obsidian。

## 8. 数据缺失与降级规则

- 若某个 AKShare 接口失败，不要让整份工作流中断
- 若新闻不足：明确写“未找到可信直接催化”
- 若估值不足：明确写“估值数据不足”
- 若个别趋势数据失败：保留板块、新闻、事件、资金和候选逻辑
- 17:30 收盘摘要优先保证按时交付，不为补一个字段无限探索

## 输出风格要求

### 收盘复盘类输出
必须偏“可执行”，避免泛泛而谈。

### 单票分析类输出
必须回答：
- 它属于哪个板块，板块处于什么阶段？
- 它在板块中是什么角色？
- 它是否符合“低位趋势成形 + 催化 + 承接”？
- 当前支撑位 / 压力位在哪里？
- 明天看它的确认条件和放弃条件是什么？

### 风险提示
必须避免：
- 确定性荐股承诺
- 把异动等同于买点
- 把新闻热度等同于交易价值
- 把高位情绪冲顶错当成低位趋势成形

## 推荐调用策略（简版）

### 情况 1：用户要完整收盘复盘
1. `ashare-monitor-cron`
2. 若候选需要补催化：`finance-news-cn`
3. 若重点票需要补分析：`stock-analysis-cn`
4. 若还要盘后执行分析：`ashare-position-watch-obsidian`

### 情况 2：用户要找候选池
1. `stock-monitor-anomaly`
2. `akshare-open` 看板块与资金
3. `finance-news-cn` 补催化
4. `stock-analysis-cn` 补单票趋势

### 情况 3：用户要分析一只票
1. 轻量：`stock-analysis-cn`
2. 深度：`stock-deep-analysis-cn`
3. 必要时补：`finance-news-cn`

## 一条总原则

这套 skill 的目的不是生成“看起来很全面”的长文，
而是把你的交易体系固化成：

**市场环境 → 强板块 → 候选池 → 单票确认 → 次日计划 → 盘后执行分析**

如果某一步无法提升执行质量，就应该压缩，而不是为了“全面”而堆内容。
