# 持仓股 / 候选股盘后分析并写入 Obsidian

当用户想要：
- 在收盘摘要之外，额外对**持仓股**和**候选股**做盘后分析
- 同时覆盖：
  - 主板股票
  - ETF / LOF 等场内基金产品
- 输出：
  - 上升/下降通道
  - 支撑位 / 压力位
  - 趋势判断
  - 明日走势预判
  - 盈亏比参考
  - 对应操作策略
- 自动保存到 Obsidian

使用本技能。

## 适用场景

适合以下需求：
1. 用户已有日常 `close-summary.md`
2. 用户会在自我复盘里写当前持仓
3. 想把“复盘”和“明日交易计划”拆成两份文件
4. 用户是短线风格，需要偏执行层的输出而非泛研究报告

## 推荐输入来源

### 1. 持仓来源
优先从最近一份包含 `当前持仓记录：` 的 `close-summary.md` 提取。

常见文本格式：
```text
当前持仓记录：白银LOF 5成（成本 3.078，2800股），康强电子 4成（成本 21.64，300股）。
```

解析时推荐正则：
```python
r'([\u4e00-\u9fa5A-Za-z0-9()（）\-]+)\s*([0-9]+成)（成本\s*([0-9.]+)，\s*([0-9]+)股）'
```

### 2. 候选股来源
优先从最近一份 `close-summary.md` 的“个股筛选 / 次日计划”提取，尤其是：
- `### 某股票（代码）`
- `所属板块：...`
- 板块阶段：`主升 / 修复 / 分歧 / 退潮 / 轮动 / 趋势`

#### 实战修正：盘后分析不能盲目锁“前一交易日”
这个任务和 09:00 / 09:26 不一样：**持仓股与候选股盘后分析通常应优先使用“今天目录”**。

更稳的顺序应是：
1. 先检查 `YYYY-MM-DD/close-summary.md` 或 `YYYY-MM-DD/latest-summary.md` 是否已生成且非空
2. 若今天目录已有收盘摘要，盘后分析基线就应直接使用今天
3. 只有今天目录还没有收盘摘要时，才回退到前一交易日目录

否则会出现：
- 收盘后已经生成了今天的 `close-summary.md`
- 但盘后分析脚本仍错误沿用昨天的复盘与候选股
- 最终导致 `data_date`、候选池、板块阶段和持仓来源全部错一天

推荐在脚本里显式加入：
- `current_day_has_close_summary()`
- `preferred_analysis_day_dir()`
- `source_validation`（expected_day / actual_day / used_fallback）

并把校验结果写到 stdout 或 context json，避免静默错取旧日期。

#### 候选股解析不要只兼容旧模板
真实环境里 `close-summary.md` 的候选区标题可能不再是：
- `## 3. 个股筛选（最重要）`

还可能升级成：
- `## 3. 个股筛选（硬过滤后）`

而且候选明细不一定都是 `### 某股票（代码）`，也可能是：
- `- 天齐锂业（002466，锂，主升，龙头）：...`
- `- 多氟多（002407，基础化工，主升，龙头）：...`

因此推荐同时兼容：
1. `个股筛选（最重要）`
2. `个股筛选（硬过滤后）`
3. `### 某股票（代码）`
4. `- 某股票（代码，板块，阶段，角色）...`
5. `## 候选：某股票（代码）`

否则脚本会只识别出持仓股，遗漏当天正式候选池。

## 关键经验

### 经验 0：盘后分析的“最新价格”必须只认当天，不能偷用前一天
这是这条链路最重要的硬规则：

1. **close-summary / holding-pnl / ledger / quote 的主基线必须锁定当天目录或当天账本**
2. 如果当天 `close-summary.md` 尚未生成：
   - 允许尝试当天 `latest-summary.md`
   - 允许尝试当天 `holding-pnl-1505.md`
   - 允许切换到其他**当天实时/收盘口径**数据源
3. **绝对不要**因为当天接口失败，就静默回退到前一天 `close-summary.md` 或前一天价格来冒充“今日盘后分析”
4. 若历史K线/基金净值未覆盖到当天，但当天实时价格可得：
   - 可以把**当天 quote 强制拼接**到历史序列末尾，再算 MA / RSI / MACD / 通道
   - 不允许直接用前一天最后一根 bar 充当今天收盘
5. 若连当天 quote 都拿不到：
   - 应明确报错或降级成“无法完成今日盘后分析”
   - 不能继续输出看似完整、实则基于前一天数据的分析

一句话原则：**多日历史可以用于趋势，最新价格必须来自今天。**

### 经验 0.1：候选股不应只看今天，建议额外融合近 3 个交易日复盘
为了让盘后分析更贴近真实短线节奏，候选股不应只从当天一份 `close-summary.md` 机械提取。

更稳的做法是：
1. 以**当天 close-summary** 为主基线；
2. 再补读**前 3 个交易日**的 close-summary / latest-summary；
3. 对候选股建立一个 `recent_history`：
   - 近 3 日出现次数
   - 出现日期
   - 近 3 日关联板块
4. 输出时对当天候选股附加：
   - 是否连续 2~3 天反复出现
   - 是否属于“今天仍在池中 + 前几天也反复出现”的延续观察标的
5. 若某只股票今天未进候选、但近 3 日里连续出现 >=2 次，且仍符合主板规则，可作为“多日跟踪补充候选”纳入分析，但必须明确标注它不是今天的新入选。

这样盘后分析就不再只是“今天这一篇”的静态截面，而是带有连续观察语义，更适合短线复盘。

### 经验 1：主板股与 ETF/LOF 要分开取数
股票与基金产品不要共用一套历史行情接口。

#### 股票
优先：
- `ak.stock_zh_a_daily(symbol='sz002119', start_date=..., end_date=..., adjust='qfq')`

实测经验：
- `stock_zh_a_daily()` 在当前环境里比 `stock_zh_a_hist()` 更稳
- `stock_zh_a_hist()` 可能出现：
  - `RemoteDisconnected`
  - `Connection aborted`
- 因此股票技术分析主路径优先选 `stock_zh_a_daily()`

#### ETF / LOF
不要直接假设 `stock_zh_a_daily()` 能拿到基金日线；在当前环境里常会失败。

更稳的拆法是：
1. **现价 / 当日涨跌 / 日内高低**：
   - LOF：`ak.fund_lof_spot_em()`
   - ETF：`ak.fund_etf_spot_em()`（如果需要）
2. **趋势 / 通道 / 支撑压力**：
   - `ak.fund_open_fund_info_em(symbol=基金代码, indicator='单位净值走势', period='1年')`

注意：
- 这意味着 ETF/LOF 的分析口径是：
  - **盈亏测算优先用场内现价**
  - **趋势与技术结构用近一年单位净值走势近似分析**
- 这种口径不等同于纯场内K线，但在接口不稳定时是可复用且可交付的方案

### 经验 2：基金名称到代码映射需要模糊处理
用户常写：
- `白银LOF`
- 但实际基金简称可能是：
  - `国投瑞银白银期货(LOF)A`
  - `国投白银LOF`

不要只做精确匹配。推荐：
1. 先用原始名称匹配 `ak.fund_name_em()` 的 `基金简称`
2. 若失败，再去掉这些后缀做二次匹配：
   - `ETF`
   - `LOF`
   - `A`
   - `C`
   - `期货`
   - 中英文括号
3. 若有多条候选，优先排序：
   - 含 `LOF` 加分
   - 含 `A` 加分
   - 含 `C` 减分

### 经验 3：候选股要补所属板块与板块阶段
如果用户明确要求候选股必须带：
- 所属板块
- 板块阶段（轮动/趋势/主升/修复等）

不要重新抓完整板块库优先归因；更稳的是：
1. 先从当日 `close-summary.md` 的 `## 2. 板块分析` 提取：
   - 板块名
   - `板块是主升、分歧、修复还是退潮：...`
2. 再从 `## 3. 个股筛选（最重要）` 提取每只候选股的：
   - 代码
   - 所属板块
3. 用代码把候选股与板块阶段映射起来

这样可保持和当天收盘摘要口径一致，不会出现“候选页和收盘页板块定义打架”的问题。

### 经验 3.2：盘后分析的候选股应分成“今天主候选”和“近3日复盘补充候选”
只把所有候选股混成一个列表，实盘价值不高。更稳的做法是：

1. **今天主候选**
   - 来自当天 `close-summary.md` / `latest-summary.md`
   - 属于当天正式进入观察池的标的
2. **近3日复盘补充候选**
   - 来自最近 3 个交易日复盘里重复出现的票
   - 仅当出现次数 >= 2 次时，才作为“补充观察上下文”纳入
   - 必须明确标注它不是“今天新入选”
3. 输出时每只票都建议带：
   - `候选来源分层：今天主候选 / 近3日复盘补充候选`
   - `近3日复盘轨迹：出现次数 / 出现日期 / 关联板块`

这样盘后分析会更接近真实短线观察池，而不是静态名单。

### 经验 3.3：可以给候选股做“多日跟踪强度分”
实战里，用户往往不只需要“有没有入选”，还需要“明天先盯谁”。

推荐增加一个评分层，对候选股按以下因素打分：
- 近 3 日出现次数
- 是否连续 2~3 天都在复盘池
- 是否属于今天主候选（若只是旧复盘补充票，应扣分）
- 当前板块阶段：主升 / 修复 / 轮动 / 退潮
- 个股角色：龙头 / 中军 / 补涨
- 明日预判：偏强 / 区间 / 偏弱
- 趋势状态：多头 / 空头
- 盈亏比
- 当日是否强势收盘

建议再映射等级：
- `S`：>= 70
- `A`：55~69
- `B`：40~54
- `C`：< 40

然后输出：
- `多日跟踪强度分：xx 分（X级）`
- `评分因子：...`
- `候选股优先级（多日跟踪强度分）`

### 经验 3.4：盘后分析最后应直接给“明日前2重点盯盘清单”
比单纯长文分析更有执行价值。推荐在文末额外增加：

```markdown
## 明日前2重点盯盘清单
1. 股票A（代码，xx分/X级，今天主候选/复盘补充）
   - 板块/阶段：...
   - 明日预判：...
   - 重点原因：...
   - 执行动作：...
2. 股票B ...
```

排序顺序建议：
1. 先按多日跟踪强度分降序
2. 今天主候选优先于复盘补充候选
3. 同分时优先主升/修复板块，再优先龙头/中军
`sector_constituent_snapshots` 的读取不能简单取“当天最新 capture_run”。实战中可能出现：
- `capture_runs` 当天有最新记录
- 但该 run 的 `sector_snapshots` / `sector_constituent_snapshots` 是空的
- 结果下游脚本误报“今日盘中未采集到 sector_constituent_snapshots”

更稳的顺序应是：
1. **优先找当天最近一条且 `sector_constituent_snapshots` 非空的 run_id**
2. 如果当天所有 run 都为空，再回退到 **全库最近一条 `sector_constituent_snapshots` 非空的 run_id**
3. 如果 DB 仍无可用梯队数据，再回退解析 `close-summary.md` 的 `## 2.1 板块梯队（来自盘中成分股缓存）`

这样即使盘中某次抓取只写入了 `capture_runs` 主记录、没有写入板块成分层，也不会让盘后分析整段梯队信息丢失。

### 经验 4：适配小资金短线，不要输出大资金分散配置逻辑
如果用户明确：
- 短线
- 资金只有约 `16000 RMB`

输出要偏向：
- 1~2 个核心标的
- 单票分两笔或试单后再补
- 先看盈亏比再决定是否出手
- 不适合输出“大资金组合分散化”风格建议

可固定加入：
- `16000 资金体量不适合同时摊太多票`
- `若开新仓，优先只做 1 个最强核心`
- `单票先用 1/2 计划仓试单，确认承接后再补`

## 推荐分析结构

### 每只标的固定输出
1. 标的名称 / 代码
2. 若是候选股：所属板块 + 当前板块阶段
3. 最新收盘 / 现价
4. 日内区间
5. 趋势判断：
   - 多头趋势
   - 空头趋势
   - 震荡 / 过渡趋势
6. 通道结构：
   - 上升通道 / 下降通道 / 横盘通道
   - 中轴 / 上轨 / 下轨
7. 均线：MA5 / MA10 / MA20 / MA60 / MA120
8. 指标：RSI14 / MACD / Signal / ATR14
9. 关键支撑位
10. 关键压力位
11. 若是持仓：成本、仓位、股数、浮动盈亏
12. 明日走势预判
13. 盈亏比参考
14. 操作策略
15. 数据备注

### 组合级结论建议包含
- 偏强跟踪
- 区间观察
- 偏弱防守
- 更适合短线小资金优先看的候选
- 对小资金的执行建议

## 技术指标与计算口径

### 1. 均线
推荐：
- `MA5 / MA10 / MA20 / MA60 / MA120`

### 2. RSI
用 `14` 周期即可：
- `RSI14`

### 3. MACD
标准：
- EMA12 - EMA26
- Signal = MACD 的 EMA9

### 4. ATR
- `ATR14`
- 用于给次日波动区间做粗估

### 5. 通道
推荐口径：
- 最近 `60` 个交易日
- 对 `close` 做线性回归
- 中轴 = 回归线末值
- 上轨 / 下轨 = 中轴 ± `2 * 残差标准差`

### 6. 支撑 / 压力
推荐：
- 近 `120` 个交易日枢轴高低点
- 再用 MA5/10/20/60/120 做交叉验证

## 数据口径提示（必须写）

若分析包含基金产品，建议在文中明确写：
- `LOF/ETF 优先使用场内最新价做盈亏测算；趋势、通道、支撑压力基于近一年单位净值走势近似分析。`

这样可以避免用户误以为基金部分是纯交易级K线结论。

## 推荐输出文件

按日期写到：
```text
/home/admin/Notes/market/ashare-monitor/YYYY-MM-DD/持仓股与候选股分析-YYYY-MM-DD.md
```

## 推荐自动化方式

推荐用 cron `script` 直接生产 Markdown，再由轻量 prompt 只负责确认写入成功。

示例：
```json
{
  "action": "create",
  "name": "ashare-position-watch-analysis",
  "schedule": "40 17 * * 1-5",
  "deliver": "local",
  "script": "ashare_position_watch_analysis.py",
  "prompt": "...只读取脚本输出 note_path 并确认文件存在..."
}
```

## 风险与限制

1. ETF/LOF 的趋势分析若依赖单位净值，不完全等同于场内K线
2. 股票与基金接口稳定性不同，不能强行共用一套数据源
3. 若 close-summary 中没有写清候选股板块，候选页的板块归因也会不完整
4. 短线预测只能做场景化预判，不能写成确定性保证

## 一条实战结论

对于“主板股票 + ETF/LOF + 小资金短线 + Obsidian 落地”的组合任务，
**最稳的路线不是追求所有品种统一接口，而是：股票用 `stock_zh_a_daily()`，基金用 `fund_lof_spot_em()/fund_etf_spot_em()` + `fund_open_fund_info_em()` 分拆处理，再把结果汇总成统一 Markdown。**
