# A 股开盘前简报（09:00 飞书版）

当用户想要：
- 每天开盘前快速看一眼 **昨夜今晨最重要的全球财经新闻**
- 将 **TrendRadar 本地输出** 作为多源新闻补充，和 Google News / RSS 一起参与盘前判断
- 同时检查 **Obsidian 中的持仓股与前一日候选股** 是否在过去 24 小时有新公告 / 大波动 / 新研报
- 并结合 **前一日复盘、前一日候选股、隔夜新闻** 生成一批 **新的候选股**
- 在正式开盘前拿到一份 **今日关注清单**，避免盘中乱操作
- 并在 **09:00 自动发送到飞书/当前会话**

使用本技能。

## 核心目标

开盘前只解决三件事：

### 1. 外面发生了什么
重点覆盖过去 12 小时内：
- 美股
- 港股
- 大宗商品（黄金 / 白银 / 原油 / 铜等）
- A 股相关政策 / 中国资本市场新闻

固定输出：
1. `【最重要3条】`：每条一句话，写清楚“发生了什么 + 可能影响什么”
2. `【A股相关】`：列出与 A 股 / 港股直接相关的新闻
3. `【一句话总结】`：市场情绪偏乐观 / 偏谨慎 / 中性

### 2. 持仓股与前一日候选股有没有动静
从 Obsidian 中提取：
- 最近一份 `close-summary.md` 的候选股
- 最近一份含 `当前持仓记录：` 的 `close-summary.md` 的持仓股

检查以下三类异常：
- 过去 24 小时内发布过公告
- 最近一个交易日涨跌幅绝对值超过 3%
- 过去 24 小时内有新的券商 / 机构研报或评级

输出原则：
- **只列异常**
- 没异常就写 `无异常`

### 3. 基于复盘与新闻生成新的候选股
必须结合以下三类上下文：
- 前一日复盘里提到的强板块 / 次日计划板块
- 前一日候选股及其所属板块
- 隔夜新闻（含 TrendRadar 多源补充）

目标不是简单重复前一日候选股，而是：
- 在强板块中筛出 **新的主板候选股**
- 必要时允许少量保留高质量沿用候选，但要显式标注“新候选 / 沿用候选”
- 对每只新候选进一步补充：
  - **板块角色**：龙头 / 中军 / 补涨 / 跟风
  - **计划关注点**
  - **确认条件**
  - **放弃条件**

### 4. 今日计划
基于：
- 隔夜新闻
- 持仓 / 旧候选异动
- 新生成的候选股

除了 `【今日关注清单】` 外，还应输出 `【今日操作计划】`，至少包括：
- `【持仓股操作】`：每只持仓给出操作标签（如：偏买 / 偏卖 / 可做T / 观察等待）、买点/加仓条件、卖点/减仓条件、放弃条件
- `【候选股操作】`：每只候选给出操作标签（如：偏买 / 先观察 / 放弃 / 重点盯）、买点/参与方式、卖点/退出方式、放弃条件
- `【今天需要特别注意】`：明确提醒今天重点看什么、注意什么
- `【今日执行计划表（可下单版）】`：必须明确列出“候选股是否应该买入、持仓是否应该卖出、买卖点、止损点位、止盈点位、短线支撑/压力、中线支撑/压力、计划持有周期（默认不超过 20 天）”，并进一步细化为：
  - `计划仓位比例`
  - `首笔试仓比例`
  - `二次加仓条件`
  - `开盘前 5 分钟动作`
  - `开盘后 15 分钟动作`
  - `高开应对 / 平开应对 / 低开应对`
  - `明确买入区 / 明确卖出区 / 止损位`
  - `竞价阈值`
  - `开盘3分钟 / 开盘5分钟 / 开盘15分钟动作`
  - `第一次卖出比例 / 第二次卖出比例`
  - `撤单/取消条件`
  - `开盘情景分级`（高开0~2%、高开2~4%、高开4%以上、平开、低开0~2%、低开2%以上）
  - `买点类型分层`（回踩买、突破买、弱转强买、只观察不买）
  - `今日市场环境与总策略`
  - `今天是否应该买入 / 是否应该卖出 / 是否空仓等待`
  - `建议最大总仓位`
  - `仓位档位规则`（退潮 / 轮动 / 修复 / 主升 / 极强共振日）
  - `从空仓到轻仓 / 从轻仓到半仓`
  - `即使看好也不能出手 / 满仓重仓保护`

生成 `【今日关注清单】`：
- `【重点盯】`：1~2 个今天必须关注的标的/线索
- `【备选观察】`：1~2 个次优先观察对象
- `【今日避坑】`：提醒今天不要做什么

候选股数量控制：
- 候选股不应超过 4 个
- 宁可少而精，也不要在盘前给出过多分散注意力的标的

## 推荐实现方式：脚本优先，Agent 只负责投递

最佳实践不是让 cron 在 09:26 临时自由搜索，而是：

1. 先用固定脚本生成本地文件
2. 再由 cron 轻量读取并发送到飞书

推荐脚本：
```text
/home/admin/.hermes/scripts/ashare_opening_brief.py
```

推荐输出文件：
```text
/home/admin/Notes/market/ashare-monitor/YYYY-MM-DD/opening-brief.md
/home/admin/Notes/market/ashare-monitor/YYYY-MM-DD/opening-brief-context.json
```

## 当前推荐的数据源

### 新闻层
优先使用多源新闻聚合：
- Google News RSS 搜索（过去 12 小时）
- TrendRadar 本地输出（hotlist + RSS SQLite）

建议按四个 bucket 抓取：
- 美股 / 美联储
- 港股 / 恒生
- 大宗商品
- A 股政策 / 中国资本市场

TrendRadar 只作为多源补充层，优先消费其本地 SQLite：
- `/home/admin/Notes/market/trendradar-output/news/*.db`
- `/home/admin/Notes/market/trendradar-output/rss/*.db`

### 个股异常层
优先使用 AkShare：
- 公告：`stock_notice_report(symbol='全部', date=...)`
- 研报：`stock_research_report_em(symbol=代码)`
- A 股日线涨跌：`stock_zh_a_daily(symbol='sh600000'/'sz000001', adjust='qfq')`
- ETF/LOF spot 兜底：`fund_lof_spot_em()` / `fund_etf_spot_em()`

### Obsidian 数据来源约定

默认按当前用户环境：
- Vault 根目录：`/home/admin/Notes`
- 市场目录：`/home/admin/Notes/market/ashare-monitor`

持仓与候选股来源约定：
1. **优先锁定“昨天目录”**，不要跨天直接拿“最近一个 close-summary”
2. 候选股优先来源：
   - 昨天目录下的 `close-summary.md`
   - 若昨天没有 `close-summary.md`，回退到昨天的 `latest-summary.md`
   - 若昨天目录仍缺少候选明细，再回退到最近一份真正存在的 `close-summary.md` 只做候选补全
3. 持仓优先来源：
   - 昨天目录下的 `持仓股与候选股分析-*.md`
   - 其次昨天目录下的 `holding-pnl-1505.md`
   - 最后才回退到 `close-summary.md`

> 这样可以复用现有收盘复盘链路，不需要用户额外维护一套晨会 watchlist 文件。

### 一个重要经验：不要把“最新文件”误当成“上一交易日文件”

真实环境里，经常会出现：
- 昨天目录已经生成了 `latest-summary.md`
- 但 `close-summary.md` 尚未落地，或者该日根本没有这个文件
- 若脚本只写“取最近一个 `close-summary.md`”，就会直接退回前天，导致：
  - `复盘来源` 显示前天
  - 候选池、持仓、板块统计全部错位一天

因此开盘前脚本必须先做：
1. 先确定“昨天目录”
2. 再在昨天目录内按优先级挑文件
3. 只有某类信息昨天确实缺失时，才做局部回退

不要用“全局最新匹配文件”替代“昨日复盘基线”。

## 定时任务建议

推荐 cron：
- 09:00 盘前简报：`0 9 * * 1-5`
- 09:26 集合竞价结束后操作表：`26 9 * * 1-5`
- deliver：`origin`
- script：
  - 09:00：`ashare_opening_brief.py`
  - 09:26：`ashare_opening_action_table.py`

推荐 prompt 应尽量轻：
- 09:00 任务只读取脚本生成的 `opening-brief.md`
- 09:26 任务只读取脚本生成的 `opening-action-table-0926.md`
- 如文件非空，则原样发给用户
- 不要在 cron 里二次展开重型搜索
- 若脚本失败，再根据脚本 stdout/stderr 给出简短失败说明

## 09:26 集合竞价结束后的操作表（新增实战链路）

如果用户不仅要 09:00 盘前简报，还要在 **集合竞价结束后（09:26）** 再收到一版更贴近实盘执行的操作表，应采用 **脚本二段式**：

### 目标
09:26 这份内容不是重新生成一篇盘前研报，而是把以下信息压缩成一张“动作排序表”：
- 09:00 开盘简报的结论
- 今日总策略 / 建议最大总仓位
- 当下开盘价 / 今开 / 昨收 / 开盘幅度
- 当前持仓账本
- 候选股执行表

### 推荐脚本
```text
/home/admin/.hermes/scripts/ashare_opening_action_table.py
```

### 推荐输出文件
```text
/home/admin/Notes/market/ashare-monitor/YYYY-MM-DD/opening-action-table-0926.md
/home/admin/Notes/market/ashare-monitor/YYYY-MM-DD/opening-action-table-0926-context.json
```

### 推荐数据拼装方式
1. **读取 09:00 盘前简报产物**：
   - `opening-brief.md`
   - `opening-brief-context.json`
2. **从 `opening-brief.md` 中解析**：
   - `今日总策略`
   - `建议最大总仓位`
   - `### 【持仓执行计划表】`
   - `### 【候选执行计划表】`
3. **从账本读取持仓基线**：
   - 优先 `ashare_ledger_lib.latest_report_summary()`
   - 持仓明细优先 `load_snapshot_rows(TODAY)`；若当天无快照，再退回 `load_latest_snapshot_rows()`
4. **09:26 开盘价抓取**：
   - 个股优先 `ak.stock_bid_ask_em(symbol=代码)`，读取 `今开 / 昨收 / 最新 / 涨幅`
   - ETF/LOF 用 `ak.fund_lof_spot_em()` / `ak.fund_etf_spot_em()`，列名注意是 `开盘价 / 昨收 / 最新价 / 涨跌幅`
   - 指数可用 `ak.stock_zh_index_spot_em()` 做一个轻量开盘环境快照
5. **生成动作表**：
   - 持仓：给出 `9:26建议`（如先减仓防守 / 先看修复 / 按计划持有做T）
   - 候选：给出 `9:26建议`（如按原计划观察 / 接近突破位等放量确认 / 高开过多不追 / 弱于预期默认取消）
   - 强调 09:26 只做排序，不抢第一笔；真正下单默认延后到 9:30 后 3-5 分钟确认

### 实战经验 / 坑点
- `stock_bid_ask_em()` 可直接拿到 A 股的 `今开`，适合 09:26 快速判断。
- ETF/LOF 不能复用 A 股字段名；实测 `fund_lof_spot_em()` 用的是 `开盘价`，不是 `今开`。
- `fund_lof_spot_em()` / `fund_etf_spot_em()` 会打印进度条，脚本里要用 `contextlib.redirect_stdout/redirect_stderr` 压掉，否则 cron stdout 会很脏。
- 09:26 脚本应该只做少量标的抓取（持仓 + 候选），不要全市场扫描。
- 若账本当天还没生成快照，允许回退到最近一个交易日的账本摘要，用于“当前是否有持仓”的判断。
- 输出重点是 **动作排序**，不是重新写一份长文分析。

## 一个重要优化建议

开盘前任务的目标不是“做一份研究报告”，而是“帮用户在 09:26 前完成注意力排序”。

因此：
- 新闻只保留最重要的 3 条
- 异常只列有变化的票
- 今日计划只给 1~2 个重点、1~2 个备选、1~2 条避坑
- 宁可短而稳，也不要长而散

## 风险与限制

1. 公开新闻聚合源存在转载与重复，必须尽量去重
2. 开盘前 09:26 有些资讯尚在快速变化，结论更适合做“开盘前预判”，不适合当成盘中机械指令
3. 研报 / 公告 / ETF 数据可得性不完全一致，拿不到时要明确写“未检出”而不是伪造
4. 该简报用于盘前整理，不构成投资建议

## 性能与超时避坑（实战补充）

在真实 cron 环境中，`ashare_opening_brief.py` 很容易因为 **AkShare 慢接口串行调用** 超过 Hermes cron 的 script 超时窗口（曾实测报错：`Script timed out after 120s`）。

盘前 09:00 任务要遵循这条原则：

### 绝对不要在 09:00 cron 脚本里做这些重操作
- 对候选板块逐个调用 `ak.stock_board_industry_cons_em(...)`
- 对观察池逐票调用 `ak.stock_research_report_em(...)`
- 对观察池逐票调用历史日线接口做趋势/均线计算
- 在同一次脚本执行里混合大量公告、研报、历史K线、板块成分抓取

这些接口在盘前时段非常容易累计到几分钟，导致：
- 今日 `opening-brief.md` 未生成
- cron 虽触发，但最终只会给用户发送失败说明

### 09:00 盘前任务的推荐降级策略
优先级应当是：
1. **准时送达**
2. 信息足够用
3. 再追求更丰富的异动与候选分析

因此盘前脚本应优先采用：
- **本地复盘文件 + Google News + TrendRadar** 生成简报主体
- 候选股优先依据：
  - 前一日复盘里的计划板块
  - 前一日候选股及其所属板块
  - 隔夜新闻推断出的优先方向
- 若来不及做逐票异动检查，可直接降级为：
  - `持仓股与前一日候选股异动：无异常` 或 `未做盘前重型检查`
  - 明确这是“时效优先”的轻量版本

### 更稳的工程实现
如果仍需要公告 / 研报 / 价格异动 / 板块成分级别的信息：
- **不要在 09:00 cron 现场抓**
- 改成前一晚或更早的后台脚本预抓取并写入本地 cache / json
- 09:00 任务只读取本地缓存并生成 Markdown

### 解析规则补充（实战踩坑）
开盘前脚本不要只兼容单一 Markdown 模板，至少要同时支持下面几种来源格式：

#### 多来源复盘文本不要先拼接后整体解析
真实环境里，开盘前上下文经常来自三份不同文件：
- 昨天目录下的 `latest-summary.md`
- 昨天目录下的 `持仓股与候选股分析-*.md`
- 用于补全的最近一份 `close-summary.md`

一个关键坑点是：**不要先把三份文本 `join` 成一个大字符串，再用 `re.search()` / `split()` 从头找第一个板块区块、候选区块、计划板块区块。**

这样很容易被较早日期的 fallback `close-summary.md` 抢到“第一个匹配”，最终导致：
- `复盘来源` 看起来像昨天，但实际解析出的板块/候选来自前天
- `板块阶段统计` 变成全 0 或错位一天
- 候选池退化成只剩持仓股，遗漏昨天复盘中的正式候选

推荐做法：
1. 先锁定昨天目录内的主来源文件
2. 对 `close_text / analysis_text / fallback_close_text` **分别解析**：
   - `candidate_sector_map`
   - `board_stage_map`
   - `plan_sectors`
3. 再按优先级合并：
   - 昨天目录内文件优先
   - fallback 只补缺失字段，不覆盖昨天已有结果
4. 输出时同时写清：
   - `复盘来源`
   - `持仓来源`
   - 若用了 fallback，可在 context json 里额外记录 `fallback_close_path`

一句话原则：**fallback 只能补洞，不能反客为主。**

1. **板块阶段**
   - `close-summary.md` 里的板块分析正文格式
   - `持仓股与候选股分析-*.md` 里的标题格式，例如：
     - `### 医疗研发外包（轮动）`
     - `### 氮肥（主升）`
   - 若只解析第一种格式，就会出现：
     - `板块阶段统计：主升 0 / 修复 0 / 轮动 0 / 分歧 0 / 退潮 0`

2. **候选股**
   - 传统格式：`名称（代码）`
   - 收盘摘要分层列表格式：
     - `- 豪威集团（603501，待补充，待确认，待判断）`
     - `- 维科技术（600152，待补充，待确认，待判断）`
   - 分析文件标题格式：
     - `## 候选：豪威集团（603501）`
   - 若只用全局宽泛正则提取所有 `名称（代码）`，容易把持仓、梯队股、无关正文一起误抓进候选池；应优先限制在“个股筛选 / 候选”相关区块内。

3. **持仓**
   - 遇到 `当前持仓记录：无。` 时要明确返回空持仓，不能继续沿用更早日期的旧持仓。

### 推荐缓存项
- 股票名称 → 代码映射
- 基金名称 → 代码映射
- 前一晚候选股公告/研报摘要
- 板块候选池或板块成分快照

### 一个经过验证的可行方向
如果 09:00 任务已经不稳定，应先把它收缩成：
- 新闻层：Google News + TrendRadar
- 候选层：沿用前一日复盘候选 + 板块优先级重排
- 异动层：允许暂时关闭重型逐票检测

这样通常可把脚本耗时从数分钟压到几秒级，再逐步把重信息迁移到预计算链路。

## 一条实战原则

对于开盘前场景，**最有价值的不是更全面，而是更可执行**：
- 先看外部情绪
- 再看自己手里的票
- 最后只留下当天最该盯的 1~2 个方向
