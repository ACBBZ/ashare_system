# A 股 09:26 开盘操作表

适用场景：
- 用户已经有 09:00 开盘简报
- 希望在 **09:26（集合竞价结束、正式开盘后）** 再收到一份更接近实盘执行的操作表
- 操作表必须结合：
  - 持仓股
  - 原候选股
  - 竞价/开盘数据
  - 大盘开盘情况
  - 开盘强势板块
  - 动态补充的新候选股
- 且输出必须明确到 **价格区间**，例如：`4.55~4.60 可买入`

## 核心原则

09:26 任务不是重复 09:00 盘前简报，而是把盘前计划升级为 **开盘后可执行动作表**。

必须先回答 4 个问题：
1. 大盘开出来强还是弱？
2. 原计划板块今天开盘有没有兑现？
3. 持仓/旧候选是强化了还是证伪了？
4. 有没有比旧候选更强的新主板候选？

## 强制数据输入

### 0. 候选股回退规则（新增）
若 `### 【候选执行计划表】` 中没有可解析代码（例如 09:00 简报因为 **A层为空** 只写了“暂无A层高置信候选”占位行），**不能**直接判定“未提取到原候选股”。

必须继续回退读取：
1. `## 3. 基于前一日复盘 + 新闻生成的新候选股` 小节；
2. `opening-brief-context.json` 中的 `new_candidates` / `focus`；
3. 必要时再结合前一日 `close-summary.md` 补齐板块、阶段、角色。

工程实现上建议：
- 先尝试从正文 `## 3 ...` 小节恢复完整候选；
- 若正文解析失败，再从 `opening-brief-context.json` 恢复；
- `new_candidates` 往往只有 `名称(代码)`，字段较弱；`focus` 常带 `代码 + 板块 + 角色`，应优先与 `new_candidates` **按代码合并**，避免只恢复出代码/名称却丢失板块与角色；
- 若 context 仍缺字段，再用前一日 `close-summary.md` / `持仓股与候选股分析-*.md` 补齐。

> 这是实战修正：09:00 表格只承载 A 层可下单计划，但 09:26 仍然需要展示 **B层/观察层候选股** 的竞价与开盘数据，不能因为 A 层为空就把候选股整段清空。另一个实战发现是：`opening-brief-context.json` 可以兜底恢复候选，但若不把 `focus` 与 `new_candidates` 合并，恢复结果经常只剩代码/名称，信息不够可执行。

### 1. 09:00 开盘简报
优先读取：
- `opening-brief.md`
- `opening-brief-context.json`

用途：
- 继承盘前情绪、总策略、最大仓位
- 读取原候选执行表
- 把 09:00 的计划映射到 09:26 的实盘动作

### 2. 账本持仓
优先读取账本，而不是老的 close-summary 文本：
- `ashare_ledger_lib.load_snapshot_rows(TODAY)`
- 如当日无快照，再回退 `load_latest_snapshot_rows()`
- 再配合 `latest_report_summary()` 读累计盈亏和“当前持仓记录”

#### 一个关键兜底
若账本当天无快照，不要立刻沿用更早交易日的持仓明细。

应先检查昨天目录下的：
- `holding-pnl-1505-context.json`

若其中：
- `holding_count == 0`
- 或 `holdings_line = 当前持仓记录：无。`

则应明确把 09:26 任务视为 **空仓基线**，只允许保留累计盈亏摘要，不能继续把前一天更早的持仓股带进“持仓股开盘计划”。

> 这是实战修正：否则会出现账本日报已经显示“昨日空仓”，但 09:26 操作表仍错误展示更早日期的持仓股。

### 3. 最近一份前一日复盘基线
不要简单写成“取最近一个 `close-summary.md`”。正确做法是：

1. **先锁定昨天目录**
2. 再在昨天目录内按优先级读取：
   - `close-summary.md`
   - 若不存在则回退 `latest-summary.md`
3. 板块 / 候选补充信息再优先读取昨天的：
   - `持仓股与候选股分析-*.md`
4. **新增实战修正**：如果昨天目录里只有 `latest-summary.md`，但它只是盘中监控快照、**不含结构化候选区**（如没有 `## 3. 个股筛选` / `## 候选：...`），则不能把它当成唯一候选基线；应继续向前回退到最近一个真正含结构化候选区的 `close-summary.md` 作为候选池补充来源，同时保留昨天分析文件做最近一日修正。
5. 只有昨天某类文件确实不存在，或昨天文件存在但**结构上不足以提供候选信息**时，才允许回退到更早日期做局部补全

用途：
- 提取板块分析
- 提取原始候选股及其所属板块 / 板块阶段 / 板块地位
- 补充昨天分析文件里的候选、板块阶段与趋势信息

> 实战踩坑：如果脚本只匹配“最近一个 `close-summary.md`”，而昨天目录只有 `latest-summary.md` 没有 `close-summary.md`，脚本就会错误退回前天，导致整个 09:26 操作表错位一天。

## 推荐行情来源

### 个股 / 持仓 / 候选竞价与开盘
股票优先：
- `ak.stock_bid_ask_em(symbol=代码)`

重点读取：
- `最新` 作为竞价/当前价近似参考
- `今开`
- `昨收`
- `buy_1` / `sell_1`
- `buy_1_vol` / `sell_1_vol`
- `最高` / `最低`

基金/LOF/ETF：
- `ak.fund_lof_spot_em()`
- `ak.fund_etf_spot_em()`

### 大盘
优先顺序建议：
1. `ak.stock_zh_index_spot_em()`
2. 若 EastMoney 接口异常或缺少 `今开/昨收`，立即回退 `ak.stock_zh_index_spot_sina()`
3. 若实时接口仍失败，继续回退本地 `ashare_monitor.db` 的 `index_snapshots` 最新批次，并优先从 `raw_json` 恢复 `今开/昨收/最新价/涨跌幅`
4. 禁止把 `sh000001` 这类指数代码简单去前缀后当成普通股票 `000001` 再查个股行情，否则会把上证指数误读成平安银行等个股数据。

至少抓：
- 上证指数
- 深证成指
- 创业板指（即使用户不做创业板，也用于风格判断）

> 这是一次关键修正：指数 fallback 必须保持 `sh/sz` 前缀语义，且要能拿到 `今开/昨收`，否则 09:26 会出现“当前涨跌幅有值，但今开/昨收全是 —”的假完整结果。

### 开盘强势板块
- `ak.stock_board_industry_name_em()`
- 若实时接口临时失败，回退到本地 `ashare_monitor.db` 最近一次 `sector_snapshots`，并在上下文里标记数据来源为回退值，避免整段输出“强势板块数据抓取失败”。
- **新增实战修正**：无论实时还是 DB 回退结果，都要对 `证券Ⅱ/证券Ⅲ`、`航海装备Ⅱ/Ⅲ` 这类同族板块做**规范化去重**，按最新批次 + 强度分只保留一个，避免 09:26 输出被重复板块挤占。

推荐过滤和排序规则：
- 先过滤：`上涨家数 >= 下跌家数`
- 且满足：`板块涨幅 >= 1.0%` **或** `领涨股涨幅 >= 8.0%`
- 再计算强度分：
  - `板块涨跌幅 * 12`
  - `+ (上涨家数 - 下跌家数) * 0.8`
  - `+ 领涨股票涨跌幅 * 2`
  - `+ 换手率`
- 按强度分排序取前 5~6 个

> 实战发现：若不过滤，容易把“上涨/下跌接近、板块本身很弱、只是单个 20cm 龙头抬分”的板块误判成 09:26 强势板块。

### 强势板块成分股 → 新候选股
- `ak.stock_board_industry_cons_em(symbol=板块名)`

推荐筛法：
- 只保留主板股票：`600/601/603/605/000/001/002/003`
- 排除创业板/科创板/北交所
- 按 `涨跌幅 + 成交额 + 换手率` 综合排序
- 对已存在于持仓/原候选里的代码去重
- 每个强势板块最多取 4~5 只候选，再全局排序后保留 4 只左右

## 竞价强弱判断口径

优先使用委托量：
- 若 `buy_1_vol / sell_1_vol >= 1.8` → `竞价偏强`
- 若 `<= 0.7` → `竞价偏弱`

若无量比，则退化到开盘幅度：
- `open_pct >= 2%` → `竞价偏强`
- `open_pct <= -1.5%` → `竞价偏弱`

否则：
- `竞价中性偏强 / 中性偏弱`

## 价格区间生成规则

### 原候选股
目标：不要只写“观察/偏买”，而要给到 **买点区间 / 突破加仓位 / 卖出区 / 止损位**。

可复用规则：
- 若已有盘前执行表中的 `低吸区` / `突破确认区` / `止损` / `止盈`，优先沿用
- 若当前价高于开盘价：
  - 买入区 = `open + 35%~70% 的日内启动段`
- 若当前价低于开盘价：
  - 买入区 = `日内支撑附近 ±0.5%`
- 若已有 `breakout`，则要求：
  - 买入区上沿不得高于 `breakout * 0.998`

> 这是一次实战修正：如果不限制 `buy_high < breakout`，会出现“买入区高于突破位”的荒谬结果，导致输出不可执行。

### 新候选股
无现成执行表时，可用简化规则：
- `buy_low/buy_high` 根据 `今开、最新、最低` 动态推导
- `stop` 设为 `日内弱支撑` 或 `min(今开, 昨收, 最低) * 0.985`
- `sell1/sell2` 设为：
  - `max(突破位, 当前价 * 1.02)`
  - `max(sell1 * 1.02, 当前价 * 1.04)`

输出时必须写成：
- `X.XX~Y.YY 买入；A.AA~B.BB 卖出；止损 Z.ZZ`

## 09:26 操作分类建议

### 持仓股
给出：
- `先减仓防守`
- `先处理风险`
- `冲高兑现一部分`
- `按计划持有/做T`

### 原候选股
给出：
- `可列入首笔观察`
- `接近突破位，等确认`
- `按计划观察`
- `观察为主`
- `高开过多，不追`
- `弱于预期，暂不买`

### 新候选股
默认更保守：
- 允许进入观察池
- 允许给明确价格区间
- 但不要因为它是“新发现”就激进高优先级开仓

## 市场总策略动态修正

在 09:26 必须允许对 09:00 的总策略做修正：
- 若上证指数当前跌幅明显扩大（如 `<= -0.8%`），则把策略降级为：
  - `轻仓试错或空仓等待`
  - 最大总仓位降到 `10%-20%`
- 若强势板块明显有延续，且盘前情绪不是“偏谨慎”，可升级为：
  - `强板块开盘有延续，可围绕最强方向做回踩确认，不追一致性高开的后排`

## 增强版实战输出（推荐）

如果用户希望“看一眼就知道今天怎么下手”，在表格中增加一列：
- `是否立即动作`

推荐统一映射为：
- `立即卖/减仓`
- `可小仓试单`
- `等回踩/等确认`
- `只观察`
- `放弃/取消`
- `持有观察`

这样 09:26 的价值就从“给信息”升级成“直接给行动优先级”。

## 再增强一层：推荐下单顺序

如果用户还希望更像盘中执行清单，建议在 `## 0. 一眼执行结论` 后再加：

```markdown
## 0.1 推荐下单顺序
1. 第一优先：华电新能，6.33 附近回踩试单；止损 6.36
2. 第二优先：泸天化，5.15~5.30 低吸，不追高，突破 5.43 再确认
3. 第三优先：节能风电，只观察，不下单；只有回到买入区且承接转强再评估
4. 禁止出手：华电辽能，今天先放弃
```

### 推荐排序规则
优先级建议按下面顺序排：
1. `可小仓试单`
2. `等回踩/等确认`
3. `只观察`
4. `放弃/取消`

### 输出要求
每条至少写清：
- 标的名 + 代码
- 来源：原候选 / 新候选 + 板块
- 动作动词：试单 / 低吸 / 观察 / 放弃
- 明确买入区或观察条件
- 止损参考

这样用户在 09:26 打开消息时，不需要自己从表格里二次排序，直接按 1、2、3 的顺序执行即可。

## 工程实现建议（新增）

如果 09:00 / 09:26 / 盘后分析脚本都需要“找前一交易日目录 + 选文件”，不要在每个脚本里各写一套。

推荐把下面逻辑抽到共享工具层：
- 列出 `ROOT/20*-*-*` 的交易日目录
- 先选“昨天目录”
- 在昨天目录内做 `close-summary.md -> latest-summary.md` 的优先回退
- 统一提供：
  - `preferred_review_day_dir(...)`
  - `pick_first_existing(...)`
  - `latest_review_file(...)`

这样可以避免：
- 09:00 用昨天
- 09:26 又退回前天
- 盘后分析再取成全局最新文件

即同一条链路里三个脚本对“前一日基线”的定义不一致。

## Cron prompt 建议

cron prompt 不要自己重做分析，只做发送层：
- 从脚本 stdout 读取 `note_path`
- 若文件存在则直接原样发送
- 若失败则明确报错并附 stdout 关键信息

## 风险与限制

1. 09:26 数据很接近开盘，价格变化快，操作表本质上是“开盘后初始预案”，不是机械指令。
2. 有些标的会出现“竞价强、开盘弱”或“开盘弱、瞬时翻强”，因此输出要保留 `等确认` / `观察为主` 这种动作层级。
3. 新候选股应偏保守，优先考虑主板、前排、成交额可承接的标的。
4. 明确价格区间比空泛建议更有用，但也必须与当下结构一致，不能生成买入区高于突破位这类无效计划。
